# 5. デプロイ手順（GitHub Actions / GitHub Pages）

## 要件
- JPX 公開 `data_j.xls` を毎日 1:00（JST）に取得
- 取得したデータから `api/17/*.json` / `api/33/*.json` を生成
- GitHub Pages へ静的ファイルとしてデプロイ

## スケジュール（毎日 1:00 JST）
GitHub Actions の cron は UTC 指定です。JST は UTC+9 なので、
- JST `01:00` は UTC `16:00`（前日）になります

推奨 cron:
- `0 16 * * *`

## ワークフロー全体の流れ（実装ガイド）
1. `actions/checkout`
2. `data_j.xls` をダウンロードして作業ディレクトリへ保存
3. データ変換（生成スクリプト）
   - `data_j.xls` から
     - 17 業種コードごと
     - 33 業種コードごと
     - の証券コード配列を作成し
   - `api/17/{code}.json` / `api/33/{code}.json` を生成
4. 生成物をコミット（Pages のデプロイ方法によりどこへコミットするかが異なる）
   - 例:
     - Pages が `gh-pages` ブランチを参照する場合: `gh-pages` へコミットしてデプロイ
     - Pages が `main`/`master` 直参照の場合: `api/` 配下を同ブランチへコミット
5. GitHub Pages のデプロイアクション実行

## Pages デプロイ方法（選択肢）
現時点の仕様書では運用方法を固定しませんが、実装時に次のどちらかを選びます。
- `gh-pages` ブランチ運用（一般的）
  - ジェネレータが `api/` を `gh-pages` に更新
- `docs/` ディレクトリ運用
  - `docs/api/` に生成して `docs` を Pages ソースにする

どちらを選ぶ場合でも、公開URL（`.../api/17/{code}.json` 等）が期待通りになるように出力パスを合わせます。

## 失敗時の扱い
- 変換スクリプトが列特定に失敗した場合（ヘッダが変わった等）は失敗させる
- 失敗時に以前の生成物を壊さないため、生成物の上書きはブランチ切替後または一時ディレクトリ経由で安全に行う方針を推奨

## 手動実行
- デバッグや確認のために `workflow_dispatch` を有効化することを推奨
